# Лабораторная работа 8

## Непрерывная интеграция с помощью Github Actions

## Цель работы

Целью данной лабораторной работы было изучение настройки непрерывной интеграции с помощью Github Actions. Также в ходе работы нужно было создать простое Web-приложение на PHP, написать для него тесты и запускать эти тесты автоматически в контейнере.

## Задание

В ходе лабораторной работы требовалось:

- создать Web-приложение на PHP;
- использовать SQLite для хранения данных;
- реализовать классы `Database` и `Page`;
- создать SQL-файл со схемой базы данных и начальными данными;
- написать тесты для всех методов классов `Database` и `Page`;
- создать `Dockerfile` для запуска проекта в контейнере;
- настроить Github Actions для автоматической сборки контейнера и запуска тестов;
- оформить отчёт по выполненной работе.

### 2. Создание Web-приложения

В папке `site/modules` был создан класс `Database`, который работает с базой данных SQLite. В классе были реализованы методы:

- `__construct($path)`
- `Execute($sql)`
- `Fetch($sql)`
- `Create($table, $data)`
- `Read($table, $id)`
- `Update($table, $id, $data)`
- `Delete($table, $id)`
- `Count($table)`

Этот класс позволяет выполнять основные операции с базой данных.

Также был создан класс `Page`, который отвечает за отображение страницы по шаблону. В нём реализованы методы:

- `__construct($template)`
- `Render($data)`

Метод `Render` подставляет значения из массива в шаблон страницы.

### 3. Создание шаблона и стилей

В папке `site/templates` был создан файл `index.tpl`, который используется как HTML-шаблон страницы.

В папке `site/styles` был создан файл `style.css`, в котором были описаны стили страницы. Это позволило сделать вывод данных более аккуратным и понятным.

### 4. Настройка конфигурации

В файле `site/config.php` были указаны настройки для подключения к базе данных SQLite. В конфигурации хранится путь к файлу базы данных.

Пример идеи конфигурации:

```php
<?php

$config = [
    "db" => [
        "path" => __DIR__ . "/database.sqlite"
    ]
];
```

### 6. Создание тестов

В папке `tests` был создан файл `testframework.php`, который содержит простой тестовый каркас. Он позволяет добавлять тесты, запускать их и выводить результат.

После этого в файле `tests/tests.php` были написаны тесты для:

- подключения к базе данных;
- метода `Execute`;
- метода `Fetch`;
- метода `Create`;
- метода `Read`;
- метода `Update`;
- метода `Delete`;
- метода `Count`;
- конструктора класса `Page`;
- метода `Render`.

Во время выполнения работы возникли ошибки с путями к файлам в контейнере Github Actions. Сначала тесты пытались подключать файлы по путям, которые были верны локально, но не совпадали со структурой внутри контейнера. Также была проблема с чтением `schema.sql`, потому что внутри контейнера этот файл уже удалялся после создания базы данных.

Эта проблема была решена тем, что тестовая база стала создаваться прямо в тестах, без зависимости от внешнего SQL-файла. Это сделало тесты более независимыми и стабильными.

### 7. Создание Dockerfile

Для запуска проекта в контейнере был создан `Dockerfile`. В нём:

- использовался PHP-образ;
- устанавливались `sqlite3` и `pdo_sqlite`;
- создавалась база данных SQLite;
- копировались файлы сайта в контейнер.

Основная идея Dockerfile состояла в том, чтобы получить готовую среду для запуска сайта и тестов.

### 8. Настройка Github Actions

В файле `.github/workflows/main.yml` была настроена автоматическая проверка проекта.

При каждом `push` в ветку `main` Github Actions выполняет следующие действия:

1. скачивает репозиторий;
2. собирает Docker-образ;
3. создаёт контейнер;
4. копирует в контейнер тесты;
5. запускает контейнер;
6. выполняет тесты через PHP;
7. останавливает контейнер;
8. удаляет контейнер.

Таким образом, после каждой отправки изменений можно автоматически проверить, работает ли проект и проходят ли тесты.

## Ответы на вопросы

### Что такое непрерывная интеграция?

Непрерывная интеграция, или CI, это практика разработки, при которой изменения в коде регулярно добавляются в общий репозиторий, после чего автоматически выполняются сборка, запуск тестов и проверка проекта. Это помогает быстро находить ошибки и не накапливать проблемы в коде.

### Для чего нужны юнит-тесты? Как часто их нужно запускать?

Юнит-тесты нужны для проверки отдельных частей программы, например функций или методов классов. Они помогают убедиться, что код работает правильно и что новые изменения не ломают старую функциональность.

Запускать юнит-тесты нужно как можно чаще:

- после написания нового кода;
- после изменения существующего кода;
- перед отправкой изменений в репозиторий;
- автоматически в CI.

### Что нужно изменить в файле `.github/workflows/main.yml` для того, чтобы тесты запускались при каждом создании запроса на слияние (Pull Request)?

Нужно добавить запуск workflow не только на `push`, но и на `pull_request`.

Пример:

```yml
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main
```

### Что нужно добавить в файл `.github/workflows/main.yml` для того, чтобы удалять созданные образы после выполнения тестов?

Нужно добавить шаг удаления Docker-образа после завершения тестов.

Пример:

```yml
- name: Remove image
  run: docker rmi containers08
```

Также можно использовать `if: always()`, чтобы шаг выполнился даже если тесты завершились с ошибкой:

```yml
- name: Remove image
  if: always()
  run: docker rmi containers08
```
